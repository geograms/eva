# NPU Transpiler Pipeline

Emits CoreML `.mlpackage`s so the runtime engine can dispatch through the
Apple Neural Engine. Triggered by `cactus convert --npu`. Runtime side:
`cactus-engine/src/model_npu.cpp`.

**Scope: audio + vision encoders only.** Text-decoder prefill is
intentionally *not* on NPU — CPU prefill is the supported path.

| Entry point | Emits | manifest key | Runtime dispatch |
|---|---|---|---|
| `run_encoder_pipeline` → `audio.py` | audio encoder | `npu_audio_encoder` | `audio_encode_via_npu` |
| `run_encoder_pipeline` → `vision.py` | vision encoder | `npu_vision_encoder` | `vision_encode_via_npu` |

`run_encoder_pipeline` reuses the exact `ComponentModuleSpec` adapter
modules + example inputs the graph transpiler captures, so the NPU encoder
stays in lockstep with the CPU component graph. Auxiliary adapter inputs
(masks, position ids) are baked into the exported model as constants —
the ANE requires a static single-input signature.

## Status

| Target | Works | Notes |
|---|---|---|
| Parakeet TDT audio encoder | **yes — verified end-to-end** | 4.5× faster warm than CPU on M-series (~155ms vs ~693ms TTFT). Use `CACTUS_ANE_AUDIO_COMPUTE_UNITS=ALL`. |
| LFM-VL vision encoder | **yes — verified end-to-end** | ~2× faster on image-specific TTFT overhead, ~4× faster on the vision encoder graph alone (the rest of TTFT is preprocessing + LM prefill, CPU-bound). |
| Gemma 4 audio + vision encoder emit | **emit verified (int8); mlpackage output matches HF reference** | Vision int8: cos_sim 0.9995 vs HF reference (MAE 0.018). Audio int8: cos_sim 0.9963 (MAE 0.0055 after applying the cactus `1/16` audio rescale, see `_gemma4_compute_native_like_audio_features`). Vision: 162 MB; audio: 282 MB. C++ runtime dispatch path is the same code that ships Parakeet ASR + LFM-VL chat — exercised end-to-end on those models in this session. Full Gemma 4 cactus runtime end-to-end requires a v2 bundle (`cactus convert`, ~15–20 GB peak), which doesn't fit on the 16 GB host where these mlpackages were produced. |
| `coremltools.optimize.linear_quantize_weights` | works | Per-component defaults: **audio int8, vision fp16**. ANE compute is fp16 internally either way; quant only affects weight storage + memory bandwidth. Audio encoders (Conformer-style) absorb int8 quant noise well — verified on Parakeet + Gemma 4. Vision encoders (ViT-style: SigLIP2, Gemma 4 vision) are more sensitive — int4 visibly corrupts Gemma 4 vision output; fp16 is the rock-solid default given the towers are small (one-shot per request, bandwidth not critical). Override via `--npu-quantize` (forces both) or `--npu-audio-quantize` / `--npu-vision-quantize` (per-component). |

## Why not NPU text prefill?

Earlier iterations also emitted a text-decoder prefill `.mlpackage`
(via the now-removed `prefill.py`). Removed because:

1. Wrapping the full text decoder (~2B params for Gemma 4 E2B) needs
   ~15–20 GB peak during `coremltools.convert`, which OOMs 16 GB hosts —
   regardless of whether the encoder emit succeeds.
2. CPU chunked prefill (post-PR #659) is fast enough that ANE prefill
   isn't a clear win for the v2 runtime.
3. `origin/main`'s NPU prefill mlpackages were pre-built externally,
   not regenerated by a transpiler — there's no precedent for v2 to
   regenerate them reliably.

The C++ runtime side (`NPUPrefill` class, `prefill_via_npu`, `load_npu_prefill`,
the `ANEPrefill` Obj-C wrapper) has been removed in lockstep.

## Known coremltools 9.0 patches

`coremltools_patches.py` carries op patches for gaps the v2 encoders hit
during conversion:

- `__and__` / `__or__` / `__xor__` — bitwise ops on bool tensors
- `new_ones` — `tensor.new_ones(shape)`
- `one_hot` — cast indices to int32 (Gemma 4 vision feeds fp32 ids)
- `unfold` — translates to `mb.sliding_windows` + a transpose so the
  window-size dim lands at the *end* (matching PyTorch semantics; MIL
  inserts it at `axis+1` by default, which breaks Gemma 4 audio's
  chunked-local attention permute chain)
- `layer_norm` epsilon dtype unification (iOS17+ enforces gamma/beta/eps
  same dtype; HF fp16-traced models hit this)
- `add_fp16_cast` MIL pass skipping `layer_norm` / `batch_norm` / etc.
- mb binop scalar-cast wrappers (sub/mul/add/div SDPA mask path).
  *Only Var operands get a `mb.cast` injected* — numpy arrays/scalars
  are retyped via `numpy.astype` so we don't accidentally materialize a
  Var in the wrong block scope when downstream passes (e.g.
  `divide_to_multiply`) call `mb.mul(..., before_op=op)` from inside a
  nested block. Mis-scoping trips the "var not visible in the block"
  validator and aborts conversion.

Add more here as new architectures expose new gaps.

## Where future NPU perf gains can come from

1. **Weight quantization (int4 / int8)** — biggest single perf knob on
   ANE. Smaller weights ⇒ less memory bandwidth ⇒ faster matmuls.
   Wired via `--npu-quantize`.
2. **SDPA preservation** — `torch.nn.functional.scaled_dot_product_attention`
   maps to a single MIL op when coremltools sees it intact. HF defaults
   to `attn_implementation="sdpa"` for compatible models, so we benefit
   automatically.
3. **Per-family adapter optimizations** — e.g., the Gemma 4 vision tower
   could probably be split (CPU patch embed + NPU transformer + CPU
   projector, matching main's `model_siglip2.cpp` pattern) for cleaner
   memory usage. Not done; whole-encoder mlpackage works today.
